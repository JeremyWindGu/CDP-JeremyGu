目标Goal

设计一种清晰、轻量、可读的“拍蚊子”交互。Design a readable and lightweight mosquito-killing interaction.

初始概念Initial Idea

从一次直接、强烈的击打开始Start from a Direct and Expressive Hit

第一版概念把拍蚊子想象成一次垂直打向投影面的强力击打，反馈也被设计得很夸张：爆裂、屏幕震动，以及类似裂屏的视觉效果。The first concept imagined mosquito killing as a strong vertical strike directly onto the projection surface. The feedback was intentionally dramatic: splatter, screen shake, and cracked-screen style effects.

这个版本在视觉气质上很明确，但也要求系统在实时环境中快速且稳定地识别一种非常特定的手势。This version had a clear visual attitude, but it also asked the interaction system to recognize a very specific gesture quickly and reliably in real time.

问题识别Problem Recognition

为什么第一版没有成立Why the First Version Didn’t Hold Up

核心问题并不在视觉想法本身，而在于表达性的概念与可读的交互状态之间并不匹配。The core issue was not the visual idea itself, but the mismatch between expressive intent and readable interaction states.

识别不稳定Recognition instability

深度通过手部关键点之间的距离来估算，但摄像头角度变化也会同时影响这些数值。Depth was estimated through the distance between hand landmarks, but camera angle changes also affected those values.

状态可读性差Poor state readability

用户很难明确判断自己的手是否已经进入可被识别的“击打”状态。Users could not clearly tell whether their hands were in a valid “hit” state before performing the action.

视觉代价过重Heavy visual cost

裂纹生成与屏幕震动提升了运算负担，也在游玩中引入了不必要的卡顿。Crack generation and screen shake increased computational load and introduced unnecessary lag during play.

概念很酷，但可用性偏弱Cool idea, weak usability

这个想法本身很有表现力，但交互闭环仍然过于模糊，难以支撑玩家稳定理解与操作。The concept felt expressive, but the interaction loop was still too ambiguous to support confident play.

原型依赖手部关键点逻辑，但状态识别对摄像头位置和角度过于敏感。Hand landmark logic informed the prototype, but state recognition was too sensitive to camera position and angle.

交互重构Interaction Redesign

把“瞄准”和“击杀”分开Separate Targeting from Killing

重构后的方案不再依赖一次直接击打，而是改成一个两阶段的流程，让系统状态更可见，也让动作更容易理解。The redesign shifted from a single direct strike to a two-step loop that made system status more visible and the action easier to understand.

01框选目标Frame target
02锁定蚊子Lock mosquitoes
03击掌Clap
04确认击杀Kill confirmed

系统不再要求玩家立刻精准击中目标，而是先允许他们框选并锁定蚊子。只有在目标被清楚选中之后，击掌才成为一个有效且明确的输入。Instead of asking players to strike the exact position immediately, the system first let them frame and lock the mosquitoes. Only after the target was clearly selected did the clap input become meaningful.

  • 降低手势识别难度Reduced gesture-recognition difficulty
  • 让当前状态对玩家可见Made the current state visible to the player
  • 建立更清晰的准备—动作反馈闭环Created a clearer feedback loop between preparation and action
  • 提升整体交互的可读性Improved the overall readability of the interaction

测试观察Playtesting

观察到的行为变成了设计证据Observed Behavior Became Design Evidence

测试不仅是验证这套交互是否可行，也暴露出一些意料之外的玩家行为，而这些行为反而让体验更具体、更可信。Testing did not just validate the interaction. It revealed unexpected player behavior that made the game feel more physical and believable.

观察到的行为Observed behavior

  • 锁定目标这一步整体较顺畅，也更容易理解。Locking the target felt relatively smooth and understandable.
  • 击掌步骤对部分玩家来说仍需要调整与重复尝试。The clap step still required adjustment and repetition for some players.
  • 玩家会自然地重复击掌、调整手部角度，甚至加入旋转或按压式动作。Players naturally repeated the clap, changed hand angle, or added a twisting / pressing motion.

设计结论Design takeaway

不完美的识别并没有完全破坏沉浸感。相反，在一些情况下，它反而生成了一种更可信的“把虫子彻底压死”的后续动作，使交互显得更有身体性。Imperfect recognition did not fully break immersion. In several cases, it actually produced a believable “finish the bug” behavior, making the interaction feel more physical rather than less successful.

内容扩展与收敛Scope Decision

谨慎扩展，再决定保留什么Expand Carefully, Then Choose What to Keep

最终方向来自对两种扩展思路的比较，并从中选择一个在短周期里更易理解、也更容易落地的方案。The final direction came from comparing possible content expansions, then selecting the one that was more legible and achievable within a short development cycle.

Proposal A

基于光的诱捕机制Light-based attraction trap

引入光作为环境诱饵，使蚊子先聚集到被照亮的区域，再进入拍击流程。Introduce light as an environmental lure so mosquitoes gather around illuminated regions before being targeted.

Proposal B

虫种与运动方式变化Bug type and movement variation

通过不同蚊虫行为与运动模式制造更清楚的变化，同时保持核心交互稳定。Different mosquito behaviors and movement patterns create clearer variety while keeping the core interaction stable.

最终选择Final Choice

最终采用 Proposal BProposal B was selected

它更直观、更容易理解，实现成本更低，也更适合短周期完成。同时，它能在保留系统稳定性的前提下，让体验层次更丰富。It was more intuitive, easier to understand, lower in implementation cost, and better suited to a short production window. It also preserved system stability while still making the experience richer.

总结Reflection

这个项目最终澄清了什么What the Project Clarified

我学到的内容What I learned

  • 比起富有表现力但模糊的输入，清晰的交互状态更重要。Clear interaction states matter more than expressive but ambiguous input.
  • 测试能暴露出值得被保留的、出乎意料的玩家行为。Playtesting can reveal unexpected player behaviors worth preserving.
  • 更强的体验有时来自对系统复杂度的主动削减。Stronger experiences can come from reducing system complexity.
  • 轻量化实现通常也会提升性能与可读性。Lightweight implementation improves both performance and readability.

这个项目体现了什么What this project demonstrates

  • 交互流程设计Interaction flow design
  • 基于手势的玩法思考Gesture-based gameplay thinking
  • 通过测试推动迭代Iteration through testing
  • 技术约束下的范围控制Scope control under technical constraints
